Load Generators Going Non-Operational

HIGH SEVERITY DAILY - MOST COMMON ISSUE
Symptoms
  • Load Generator shows "Non-Operational" status in Lab Management
  • Load Generator becomes unavailable after clicking "Reconfigure Host"
  • Load Generator fails during test execution
  • Agent icon may appear green but Load Generator still not functional
  • Error message: "Host is not operational"
  • On-premise Load Generators affected more frequently than cloud
  • Multiple Load Generators going non-operational simultaneously
  • Load Generator was working fine, suddenly fails
Root Causes
  • Test not stopped gracefully before reconfigure action
  • MI Listener communication failure or service disruption
  • Network connectivity disruption between Load Generator and MI Listener
  • Load Generator agent services unresponsive or crashed
  • Previous test cleanup incomplete leaving processes hanging
  • Disk space exhaustion on Load Generator
  • Certificate mismatch between Load Generator and MI Listener
  • Windows updates or patches affecting agent service
  • Antivirus blocking agent processes
  • Memory exhaustion causing agent crash
Resolution Steps
Step 1: First Attempt - Single Reconfigure

Success Rate: ~60% of cases resolve with single reconfigure

  1. Navigate to Lab Management > Hosts
  2. Select the non-operational Load Generator
  3. Click "Reconfigure Host" button
  4. Wait for status to change to "Operational" (typically 2-3 minutes)
  5. Check Event Log tab for any error messages
  6. If status changes to Operational, verify with small test
Step 2: Multiple Reconfigure Attempts
  1. Retry reconfigure action 2-3 more times
  2. Allow 2-3 minute intervals between attempts
  3. Success rate improves with multiple attempts
  4. Document if specific error appears in Event Log
Step 3: Check Load Generator Status Directly
# Check agent service sc query "LoadRunner Agent Service" # Check disk space wmic logicaldisk get size,freespace,caption # Check network to MI Listener ping [MI_LISTENER_IP] telnet [MI_LISTENER_IP] 443
Step 4: Manual Agent Restart
# Stop agent net stop "LoadRunner Agent Service" # Wait 30 seconds timeout /t 30 # Start agent net start "LoadRunner Agent Service"
Step 5: MI Listener Restart (Last Resort)
‚ö†Ô∏è CRITICAL WARNING: This stops ALL running tests for ALL users!

Required actions BEFORE restart:
  • Post message in team communication channel
  • Check Test Runs module for any running tests
  • Get approval from team lead if needed
  • Document reason for restart
  1. Access Jenkins self-service job: pe-ss-auto-start
  2. Select MI Listener server from dropdown
  3. Click "Stop" and wait for confirmation
  4. Wait exactly 2 minutes (important for cleanup)
  5. Click "Start" and wait for startup
  6. Verify all Load Generators return to Operational
  7. Send notification to team
Prevention Strategy:
  • Always stop tests gracefully
  • Reconfigure Load Generators BEFORE critical tests
  • Monitor Load Generator health during long tests
  • Maintain 20% free disk space minimum
  • Schedule weekly Load Generator health checks
Escalate to Core Team if:
  • Reconfigure fails after 3 attempts
  • MI Listener restart doesn't resolve issue
  • Multiple Load Generators affected simultaneously
  • Physical hardware issues suspected
  • Network infrastructure changes needed

LoadRunner Enterprise Portal Access Problems

HIGH SEVERITY WEEKLY
Symptoms
  • Portal URL completely inaccessible
  • Test Management page fails to load or shows blank page
  • UI becomes unresponsive or hangs
  • "Service Unavailable" or "503" errors
  • Login page loads but authentication fails for all users
  • Session timeout immediately after login
Root Causes
  • Database password expired (most common): Happens every 90 days
  • LoadRunner Enterprise application services not running
  • Database connection pool exhausted
  • Network/firewall blocking access on ports 8080, 443
  • Memory/CPU exhaustion on application server
  • IIS application pool crashed or stopped
Resolution Steps
Step 1: Initial Checks
  1. Check scheduled maintenance calendar
  2. Verify correct URL format
  3. Determine if only you or all users affected
Step 2: Network Connectivity Tests
# Basic connectivity ping [LRE_SERVER] # DNS resolution nslookup [LRE_SERVER_URL] # Port connectivity telnet [LRE_SERVER] 443 telnet [LRE_SERVER] 8080 # PowerShell alternative Test-NetConnection [LRE_SERVER] -Port 443
Step 3: Browser-Based Troubleshooting
  1. Try incognito/private mode (Ctrl+Shift+N)
  2. Try different browsers (Chrome, Firefox, Edge)
  3. Test from different network location
  4. Try direct IP instead of hostname
Step 4: Browser Console Diagnostics

Open Developer Tools (F12) and check Console tab for errors:

ERR_CONNECTION_REFUSED ‚Üí Server not responding ERR_NAME_NOT_RESOLVED ‚Üí DNS problem 401 Unauthorized ‚Üí Authentication issue 403 Forbidden ‚Üí Permission denied 500 Internal Server Error ‚Üí Server-side error 502 Bad Gateway ‚Üí Load balancer issue 503 Service Unavailable ‚Üí Service down
Step 5: Clear Browser Cache

Chrome:

  1. Press Ctrl+Shift+Delete
  2. Select "All time"
  3. Check: Browsing history, Cookies, Cached images
  4. Click "Clear data"
  5. Restart browser
Database Password Expiry - Most Common Cause:

Characteristics:
  • Affects ALL users simultaneously
  • Occurs every 90 days
  • Complete portal becomes inaccessible
  • Usually happens on weekends

Resolution:
  • Only DBA team can reset password
  • Core Team must coordinate with DBA
  • Services must be restarted after update
  • Typical resolution time: 1-4 hours
Escalate to Core Team if:
  • Portal inaccessible for more than 15 minutes
  • Blocking scheduled test execution
  • Multiple users affected
  • Suspected database password expiry

Test Execution Hanging

HIGH SEVERITY SEVERAL TIMES WEEKLY
Symptoms
  • Test execution hangs mid-run without errors
  • Stop button clicked but test won't stop
  • Test remains in "Stopping" state indefinitely
  • Test shows "Running" but no requests being sent
  • Vuser count frozen, no ramp up/down
  • Cannot start new tests while hung test exists
Root Causes
  • Load Generator crash during execution
  • Network connectivity loss between Controller and Load Generator
  • Controller process (wlrun.exe) unresponsive
  • MI Listener communication timeout
  • Script infinite loop or deadlock
Resolution Steps
Step 1: Initial Assessment (5 minutes)
  1. Note exact time test stopped responding
  2. Take screenshot of hung state
  3. Record Run ID from URL
  4. Check if any Vusers still generating load
  5. Verify Load Generator status in Lab Management
Step 2: Wait for Automatic Timeout (30 minutes)
Important: LoadRunner Enterprise has built-in 30-minute timeout
  • System should eventually terminate hung test
  • Do NOT click Stop repeatedly
  • Monitor for automatic termination
  • Use time to prepare escalation info
Step 3: Check Load Generator Status
  1. Go to Lab Management > Hosts
  2. Check status of all Load Generators in test
  3. If any show "Non-Operational", this explains hang
  4. Try reconfiguring non-operational Load Generators
Step 4: Check Controller Process
# Check process tasklist | findstr wlrun # Get process details wmic process where name="wlrun.exe" get ProcessId,WorkingSetSize
Step 5: Force Stop Attempt
  1. In Test Runs module, find your run
  2. Click "Force Stop" if available
  3. Wait 5 minutes for force stop to complete
Prevention:
  • Set appropriate timeout values in scenario
  • Monitor Load Generator health before long tests
  • Use smaller ramp-up increments
  • Set maximum test duration
  • Implement heartbeat transactions
Escalate to Core Team if:
  • Test remains hung after 30-minute timeout
  • Multiple tests hung simultaneously
  • Blocking other users from resources
  • Need immediate force termination (provide Run ID)

Cloud/Elastic Load Generator Provisioning Failures

HIGH SEVERITY COMMON WITH NEW REGIONS
Symptoms
  • Error: "Failed to allocate the following cloud hosts"
  • Cloud Load Generators not spinning up
  • Timeout errors during provisioning
  • "Insufficient capacity" errors from cloud provider
  • Partial allocation - some cloud LGs start, others fail
Root Causes
  • LoadRunner Enterprise cloud connector service issues
  • Cloud provider API connectivity problems
  • AWS/Azure account quota exceeded
  • VPC or subnet misconfiguration
  • AMI/image unavailable in selected region
  • IAM permissions insufficient
  • Security group rules blocking communication
  • Insufficient capacity in availability zone
Resolution Steps
Step 1: Verify Cloud Template Configuration
  1. In Lab Management > Cloud Templates
  2. Confirm correct template is selected
  3. Verify region matches requirements
  4. Check instance type availability
  5. Confirm image/AMI ID is valid
  6. Review security group settings
Step 2: Check Recent Changes
  • Any cloud template modifications in last 24 hours?
  • AWS/Azure account changes?
  • Quota adjustments or billing issues?
  • Network configuration changes?
  • Check cloud provider service health dashboard
Step 3: Retry with Alternatives
  1. Wait 5-10 minutes for transient issues to clear
  2. Try different cloud template if available
  3. Select different availability zone
  4. Reduce number of requested Load Generators
  5. Try different instance size/type
  6. Use on-premise Load Generators as fallback
Step 4: Validate Cloud Account (AWS)
# Check AWS account (if CLI available) aws ec2 describe-account-attributes aws service-quotas list-service-quotas --service-code ec2 aws ec2 describe-vpcs aws ec2 describe-subnets
Known Solutions:
  • Restart LoadRunner Enterprise cloud connector service
  • Create region-specific custom AMI
  • Increase account quotas in cloud provider console
  • Update IAM role permissions
  • Fix security group ingress/egress rules
  • Switch to different availability zone
Escalate to Core Team if:
  • Consistent failures across multiple attempts
  • New region setup required
  • Custom image creation needed
  • Cloud account configuration changes required
  • Service account permission updates needed

TruClient Test Results Missing

MEDIUM SEVERITY VERSION-DEPENDENT
Symptoms
  • TruClient test executes but results are missing
  • Results show as "Passed" but contain no data
  • Vuser live logs cannot be downloaded
  • Transaction response times show as zero
  • Summary report missing despite successful run
Root Causes
  • Known issue in LoadRunner Enterprise versions 2020-2022
  • Elastic Load Generator incompatibility with TruClient
  • Result file corruption during collection
  • TruClient engine crash not reported properly
  • Browser process termination during test
Resolution Steps
  1. Check LoadRunner Enterprise version in Help > About
  2. Try manual collation: Go to test run > "Collate Results"
  3. Switch to static/on-premise Load Generators for TruClient
  4. Check if raw results exist on Load Generator:
    C:\LoadRunner\tmp\results\[Run_ID] dir /s *.eve dir /s *.qtp
Known Limitations:
  • Vuser live logs unsupported on elastic Load Generators
  • Some TruClient versions incompatible with certain LRE versions
  • Result size limitations (>2GB may fail)
  • Browser driver updates may break compatibility
Workaround Options:
  • Use static/on-premise Load Generators exclusively
  • Enable extended log mode before test
  • Download results immediately after test completes
  • Consider using Web HTTP/HTML protocol instead
  • Implement custom logging in TruClient script

Script Upload and Deployment Problems

MEDIUM SEVERITY DAILY DURING DEVELOPMENT
Common Errors
  • Error: "Failed Downloading ALM Client"
  • "Only zipped VuGen scripts" error
  • Script uploads but won't run from Controller
  • Parameter files not found during execution
  • Script works in VuGen but fails in Controller
Root Causes
  • OTA client initialization failure
  • Incorrect ZIP file structure
  • Case sensitivity issues in file references
  • Missing data or parameter files
  • Path resolution differences VuGen vs Controller
  • Spaces in file names causing issues
Resolution: ZIP Structure
‚úÖ CORRECT structure: ScriptName.zip ‚îî‚îÄ‚îÄ ScriptName/ ‚îú‚îÄ‚îÄ Action.c ‚îú‚îÄ‚îÄ vuser_init.c ‚îú‚îÄ‚îÄ vuser_end.c ‚îú‚îÄ‚îÄ ScriptName.usr ‚îú‚îÄ‚îÄ ScriptName.prm ‚îú‚îÄ‚îÄ default.cfg ‚îú‚îÄ‚îÄ default.usp ‚îú‚îÄ‚îÄ data/ ‚îÇ ‚îî‚îÄ‚îÄ paramfile.dat ‚îî‚îÄ‚îÄ ... ‚ùå WRONG structure: ScriptName.zip ‚îî‚îÄ‚îÄ Scripts/ ‚îî‚îÄ‚îÄ ScriptName/ ‚îî‚îÄ‚îÄ ...
How to Create Correct ZIP:
  1. Navigate to PARENT folder of your script
  2. Select the script folder (not its contents)
  3. Right-click > Send to > Compressed (zipped) folder
  4. Rename ZIP to match script name exactly
  5. Verify ZIP structure by opening and checking
Resolution: Parameter File Issues

Problem: LoadRunner Enterprise is case-sensitive, VuGen is not

‚ùå In VuGen works: UserData.DAT (file) userdata.dat (reference in .prm) ‚úÖ In LRE must match exactly: UserData.dat (file) UserData.dat (reference in .prm)
Steps to Fix Parameter Files:
  1. Open ScriptName.prm file in text editor
  2. For each parameter, check:
    [parameter:UserName] Type=File TableLocation=Local ColumnName=UserName Table=UserData.dat ‚Üê Must match file name EXACTLY
  3. Remove spaces from ALL file names
  4. Change "User Data.dat" to "UserData.dat"
  5. Ensure TableLocation="Local"
  6. Save .prm file and re-ZIP
Resolution: OTA Client Failures
  1. Clear browser cache and cookies completely
  2. Try different browser (Chrome, Firefox, Edge)
  3. Check network connectivity to LoadRunner Enterprise server
  4. Verify user has upload permissions in project
  5. If persists, contact Core Team - may need OTA client reinstall
Resolution: Script Works in VuGen But Not Controller

Common causes and fixes:

1. Runtime Settings Differences
  1. In VuGen, go to Runtime Settings
  2. File > Save Runtime Settings
  3. In Controller scenario, right-click script group
  4. Edit Runtime Settings > Import from VuGen
2. Include Files Missing
// If script has: #include "custom_functions.h" // Must include custom_functions.h in ZIP // Place in script root folder
3. DLL Dependencies
  • Copy all required DLLs to script folder
  • Include in ZIP file
  • Common DLLs needed: encryption, database connectors
Best Practices for Script Upload:
  • Test upload immediately after script creation
  • Use consistent naming (no spaces, no special chars)
  • Test in standalone Controller before LRE upload
  • Create checklist: .usr, .prm, .c files, data folder
  • Version control your script ZIPs
  • Document external dependencies

Timeslot and Scheduling Issues

HIGH SEVERITY WEEKLY
Issue 1: Autostart Timeslot Didn't Start
Symptoms:
  • Timeslot was scheduled for autostart but test didn't begin
  • Timeslot shows as complete but no test execution occurred
  • No error message visible in UI
Root Causes:
  • Timeslot became invalid due to host allocation problems
  • License issues prevented test from starting
  • Performance test became invalid (script changes, errors)
  • Autostart option was not properly enabled
  • System stopped trying after autostart window expired
Resolution:
  1. Open timeslot in Timeslots module (double-click)
  2. Check "Details" tab for failure reason
  3. Verify "Automatically" option was checked when reserving
  4. Review host allocation:
    • Click "Host Details" or "Hosts" tab
    • Check if all required hosts were available
    • Note any "Not Available" hosts
  5. Verify test validity:
    • Go to Test Management
    • Open the test in design view
    • Check for any validation errors
  6. Check license availability:
    • Verify project has sufficient license allocation
    • Check if other tests consumed licenses
  7. Use "Calculate Availability":
    • In timeslot, click "Calculate Availability"
    • System recalculates if timeslot is valid
    • Shows specific reasons if invalid
Issue 2: Timeslot Became Invalid - Host Allocation
Symptoms:
  • Timeslot shows as "Invalid"
  • Error indicates host allocation failure
  • System could not allocate all requested resources
Root Causes:
  • Changes in host properties or status after timeslot reserved
  • Host assignment to pools changed
  • Pool assignment to project modified
  • Not enough available hosts matching requirements
  • Host became non-operational after reservation
Resolution:
  1. Check host status:
    • Go to Lab Management > Hosts
    • Verify all hosts used in timeslot
    • Check operational status
  2. Review Event Logs:
    • Click each host
    • Check Event Logs tab
    • Look for PC_hostChangeStateSuccess events
    • Look for PC_hostChangeStatusSuccess events
  3. Verify host pools:
    • Ensure required hosts assigned to correct pools
    • Check if pool memberships changed
  4. Check pool assignments to project:
    • Verify pools are assigned to your project
    • Check if assignments were modified
  5. Try to recalculate:
    • Use "Calculate Availability" button
    • System may reshuffle hosts
    • May find alternative host allocation
Issue 3: Test Run Failed - License Issue
Symptoms:
  • Test fails to start citing license issues
  • Timeslot became invalid due to license changes
  • Error indicates insufficient Vuser licenses
Root Causes:
  • Project limits changed after timeslot reserved
  • LoadRunner Enterprise license pool exhausted
  • License allocation changed by administrator
  • Concurrent tests consuming available licenses
Resolution:
  1. Check project limits in project settings
  2. Verify LoadRunner Enterprise license usage across projects
  3. Check if other tests are running and consuming licenses
  4. Contact administrator for license reallocation if needed
Issue 4: Performance Test Became Invalid
Symptoms:
  • Error indicates performance test is invalid
  • Test was valid when timeslot created
  • Became invalid later
Root Causes:
  • Someone modified test after timeslot reservation
  • Error in allocation of load generators to groups
  • Changes to scripts connected to the test
  • Script became unavailable or was deleted
Resolution:
  1. Go to Test Management
  2. Open the test in design mode
  3. Check for validation errors on design page
  4. Verify all scripts assigned to test still exist
  5. Check load generator allocation to script groups
  6. Fix any validation errors shown
  7. Re-save the test
  8. Verify test validates successfully
Prevention Strategies:
  • Reserve timeslots close to execution time
  • Verify test validity before reserving timeslot
  • Check license availability before scheduling
  • Communicate with team about shared resources
  • Use "Calculate Availability" before finalizing
  • Don't modify tests after timeslot reservation
Escalate to Core Team if:
  • Host allocation issues require infrastructure changes
  • License reallocation needed
  • Hosts are down and require IT intervention
  • Pool configuration changes needed

Result Collation Timeout Issues

HIGH SEVERITY COMMON WITH LONG TESTS
Symptoms
  • Error: "Failed to collate results. Reason: 'Failed to collate. Timeout exceeded'"
  • Typically occurs after long-duration tests (8+ hours, 24+ hours)
  • Collation process starts but never completes
  • Progress bar shows activity then times out
  • Test shows as "Collating" indefinitely
Root Causes
  • Default collate timeout (5 minutes) insufficient for large result sets
  • Very large test results requiring more processing time
  • Network delays between load generators and LoadRunner Enterprise server
  • Connectivity issues during collation process
  • Large number of transactions generating massive data
Resolution Steps
Step 1: Increase Collate Timeout
  1. Login to LoadRunner Enterprise at: http://[LRE_Server]:8080/qcbin
  2. Navigate to: Tools menu (top right)
  3. Select: Performance Center Controller Options
  4. Go to: Run Results > Collate Settings
  5. Find setting: "Collate timeout in minutes"
  6. Current default: 5 minutes
  7. Change to: 30 minutes (for 8-12 hour tests)
  8. Or: 60 minutes (for 24+ hour tests)
  9. Click "Apply" then "OK"
Timeout Recommendations by Test Duration:
  • 1-4 hour tests: 10-15 minutes
  • 4-8 hour tests: 20-30 minutes
  • 8-12 hour tests: 30-45 minutes
  • 12-24 hour tests: 45-60 minutes
  • 24+ hour tests: 60-90 minutes
Step 2: Retry Collation
  1. After increasing timeout, go to Test Runs module
  2. Find your failed test run
  3. Click on the run to open details
  4. Click "Collate Results" button
  5. Wait for collation to complete (may take the full timeout period)
  6. Monitor progress - don't interrupt
Step 3: Manual Collation from Controller

If automatic collation continues to fail:

  1. Download LoadRunner Controller client if not installed
  2. In Test Runs module, find the run
  3. Click "Download Results"
  4. Save raw results to local machine
  5. Open LoadRunner Controller
  6. File > Open Result File
  7. Browse to downloaded results
  8. Controller will collate locally
  9. This bypasses network timeout issues
Step 4: Check Network Connectivity
# From LoadRunner Enterprise server # Test connectivity to Load Generators ping [LOAD_GENERATOR_IP] # Check for packet loss ping [LOAD_GENERATOR_IP] -n 100 # Test large data transfer # (collation transfers large files)
Prevention is Key:

For long-duration tests (12+ hours), ALWAYS increase the collate timeout BEFORE running the test.

Do not wait for timeout failure - configure proactively!
Best Practices:
  • Increase timeout before long tests, not after failure
  • Calculate expected result size: ~1-2 MB per Vuser-hour
  • For very large tests, consider local analysis
  • Monitor network stability during collation
  • Schedule long tests when network load is low

    • Local machine may have more resources
  4. Analyze smaller time ranges:
    • If possible, analyze test in segments
    • First hour, then second hour, etc.
    • Combine analysis results
  5. Check result integrity:
    • Verify all .eve files present from all LGs
    • Check for 0-byte files
    • Verify files not corrupted

License Issues

HIGH SEVERITY OCCASIONAL
No Valid Licenses Despite License Utility Showing Active
Symptoms:
  • Controller shows "No valid licenses" error
  • License Utility shows license is active and valid
  • License suddenly stopped working after Controller was closed and reopened
  • Issue appears randomly even with valid license
Root Causes:
  • License service not running properly
  • License cache corruption
  • Communication issue between Controller and License service
  • Temporary license service glitch
Resolution:
  1. Restart License service:
    # Open Services (services.msc) # Find: LoadRunner License Service # or: Performance Center License Service # Right-click > Restart
  2. Restart Controller:
    • Close LoadRunner Controller completely
    • Wait 30 seconds
    • Reopen Controller
    • Try starting test again
  3. Verify license location:
    • Open License Utility
    • Check license file path is correct
    • Verify file exists at that location
  4. Re-apply license:
    • In License Utility, remove license
    • Re-add the same license
    • Restart Controller
  5. Temporary workaround:
    • If issue persists, may need reinstall
    • Document issue for Core Team
License Exceeded During Test Execution
Symptoms:
  • Test starts successfully but fails mid-execution
  • Error message indicates license limit exceeded
  • Some Vusers fail to initialize
  • Message: "Maximum number of Vusers exceeded"
Root Causes:
  • Other concurrent tests consuming licenses
  • Test configured for more Vusers than license allows
  • Project allocation exceeded
  • License pool exhausted across multiple projects
Resolution:
  1. Check concurrent tests:
    • Go to Test Runs module
    • Filter for "Running" tests
    • Note how many Vusers they're using
  2. Review license allocation:
    • Check total Vuser licenses available
    • Compare to requested Vusers in your test
    • Check project-level allocation limits
  3. Reduce Vuser count:
    • Modify test to use fewer Vusers
    • Run within license limits
    • Or wait for other tests to complete
  4. Schedule accordingly:
    • Coordinate with team on large tests
    • Schedule tests to avoid license conflicts
    • Use timeslots to reserve licenses
Understanding License Types:
  • Concurrent Vusers: Total Vusers running across all tests
  • Project Allocation: Max Vusers allocated to specific project
  • Protocol-specific: Some protocols have separate license pools
Escalate to Core Team if:
  • License service restart doesn't resolve issue
  • License server configuration needs review
  • Additional licenses need to be purchased
  • License allocation needs adjustment across projects

Error -10343: Communication Error - Failed to Connect

HIGH SEVERITY VERY COMMON
Full Error Message
Error -10343: Communication error: Failed to connect to remote host [hostname]
Symptoms
  • Cannot connect Controller to Load Generator
  • Agent process appears to be running but connection fails
  • Error message: "Failed to connect to remote host"
  • Load Generator shows as disconnected in Controller
Root Causes
  • Version mismatch between Controller and Load Generator
  • Network connectivity issues or firewall blocking required ports
  • LoadRunner Agent service/process not running properly
  • Multiple Network Interface Cards (NICs) causing routing confusion
  • Hostname resolution issues
Resolution Steps
  1. Verify versions match:
    • Ensure same LoadRunner version on both Controller and Load Generator
    • Check Service Pack levels - must match exactly
    • In VuGen/Controller: Help > About
    • On Load Generator: Check installed program version
  2. Check network connectivity:
    # From Controller machine, test connectivity ping [LOAD_GENERATOR_IP] # Should get replies - if timeout, network issue # Verify bidirectional connectivity # From Load Generator, ping Controller too
  3. Verify Agent is running:
    # On Load Generator: # Check if LoadRunner Agent process running # Windows: sc query "LoadRunner Agent Service" # Should show: STATE = RUNNING # If not running: net start "LoadRunner Agent Service"
  4. Check firewall settings:
    • Ensure ports 50500 and 54345 not blocked by firewall
    • Check Windows Firewall on both Controller and Load Generator
    • Check network firewall if exists between machines
    # Test port connectivity telnet [LOAD_GENERATOR_IP] 54345 telnet [LOAD_GENERATOR_IP] 50500 # If connection refused, port blocked
  5. Verify hostname resolution:
    # Check if hostname resolves nslookup [LOAD_GENERATOR_HOSTNAME] # If fails, add to hosts file: # Windows: C:\Windows\System32\drivers\etc\hosts # Add line: [IP_ADDRESS] [HOSTNAME]
  6. Check for multiple NICs:
    • If multiple network adapters exist, configure which to use
    • Open Network Connections > Advanced > Advanced Settings
    • Move primary adapter to top of list
    • Or disable unused adapters during testing
Escalate to Core Team if:
  • Issue persists after verifying all network settings and versions
  • Infrastructure changes required (firewall rules, routing)
  • Version upgrade needed to match

Error -10344: Communication Error - Failed to Bind Socket

HIGH SEVERITY COMMON
Full Error Message
Error -10344: Communication error: Failed to bind socket A process on the machine is already bound to the same address
Symptoms
  • LoadRunner Agent fails to start or cannot establish connection
  • Two Way Communication Error: Function two_way_comm_create_acceptor failed
  • Port 54345 or 50500 reported as already in use
Root Causes
  • Port 54345 or 50500 is already in use by another process
  • Previous LoadRunner process did not terminate cleanly
  • Another application is using the required ports
Resolution Steps
  1. Check port usage:
    # Windows - check which process using port netstat -ano | findstr "54345 50500" # Output shows: # TCP 0.0.0.0:54345 0.0.0.0:0 LISTENING [PID] # To see process name (requires admin): netstat -anob | findstr "54345"
  2. Identify the process:
    # Once you have PID, check what it is tasklist | findstr [PID] # Or use Task Manager: # Ctrl+Shift+Esc > Details tab > Sort by PID
  3. Stop LoadRunner Agent:
    # Stop the service net stop "LoadRunner Agent Service" # Wait 30 seconds for cleanup timeout /t 30
  4. Kill conflicting process:
    # If process is old LoadRunner process taskkill /F /PID [PID] # Or for specific process: taskkill /F /IM m_agent_daemon.exe taskkill /F /IM mmdrv.exe
  5. Restart Agent:
    # Start the service again net start "LoadRunner Agent Service" # Verify it started sc query "LoadRunner Agent Service"
  6. Reboot if necessary:
    • If ports remain occupied after killing process
    • If cannot identify or kill process
    • Reboot the Load Generator machine
    • Verify service starts automatically after reboot
Common Processes Using Ports:
  • m_agent_daemon.exe: LoadRunner Agent (expected)
  • mmdrv.exe: Old Vuser process (kill it)
  • mdrv.exe: Old Vuser process (kill it)
  • Other applications: Should not use these ports
Escalate to Core Team if:
  • Ports occupied by critical system services
  • Lack permissions to terminate processes
  • Issue persists after reboot
  • Unknown process consistently using ports
