1. Load Generator Connection Issues

Error -10343 Communication error: Failed to connect to remote host HIGH

Symptoms:

  • Cannot connect Controller to Load Generator
  • Agent process appears to be running but connection fails
  • Error message: "Failed to connect to remote host"
  • Load Generator shows as disconnected in Controller

Root Causes:

  • Version mismatch between Controller and Load Generator
  • Network connectivity issues or firewall blocking required ports
  • LoadRunner Agent service/process not running properly
  • Multiple Network Interface Cards (NICs) causing routing confusion
  • Hostname resolution issues

Resolution Steps:

  1. Verify versions match: Ensure the same LoadRunner version and Service Pack on both Controller and Load Generator machines
  2. Check network connectivity: Verify bidirectional ping between Controller and Load Generator
  3. Verify Agent is running: Check that LoadRunner Agent process or service is running on the remote Load Generator machine
  4. Check firewall settings: Ensure ports 50500 and 54345 are not blocked by firewall
  5. Verify hostname resolution: Add IP address and machine name to the hosts file on both machines if DNS resolution is problematic
  6. Check for multiple NICs: If multiple network cards exist, configure which NIC to use for communication
‚ö†Ô∏è Escalate to Core Team if: Issue persists after verifying all network settings and versions, or if infrastructure changes are required

Error -10344 Communication error: Failed to bind socket HIGH

Symptoms:

  • Error message: "A process on the machine is already bound to the same address"
  • LoadRunner Agent fails to start or cannot establish connection
  • Two Way Communication Error: Function two_way_comm_create_acceptor failed

Root Causes:

  • Port 54345 or 50500 is already in use by another process
  • Previous LoadRunner process did not terminate cleanly
  • Another application is using the required ports

Resolution Steps:

  1. Check port usage: Run netstat -an on the Load Generator and check if ports 54345 and 50500 are occupied
  2. Identify the process: Use netstat -anob (Windows) to identify which process is using the port
  3. Stop LoadRunner Agent: Stop the LoadRunner Agent service/process
  4. Kill conflicting process: Terminate any process using port 54345 or 50500
  5. Restart Agent: Start the LoadRunner Agent service/process again
  6. Reboot if necessary: If ports remain occupied, reboot the Load Generator machine
‚ö†Ô∏è Escalate to Core Team if: Ports are occupied by critical system services or you lack permissions to terminate processes

Load Generator Connected but lr_bridge Process Not Starting MEDIUM

Symptoms:

  • Agent icon turns green but connection still fails
  • lr_bridge.exe process does not appear in Task Manager on Load Generator
  • Cannot initialize Vusers on the Load Generator

Root Causes:

  • Insufficient disk space on Load Generator
  • Permissions issues preventing process creation
  • LoadRunner temporary folder issues
  • Antivirus or security software blocking process

Resolution Steps:

  1. Check disk space: Verify that the Load Generator has sufficient disk space, especially in the temp folder
  2. Check temp folder: Verify <LoadRunner Install>\temp folder is not full
  3. Clear temp files: Clean up old temporary files from LoadRunner temp directory
  4. Check permissions: Ensure the LoadRunner Agent has proper permissions to create processes
  5. Verify antivirus exclusions: Add LoadRunner directories to antivirus exclusion list

Load Generator Was Working, Now Suddenly Failing HIGH

Symptoms:

  • Load Generator connected successfully for weeks, now failing
  • No obvious changes made to configuration
  • Error: Communication error - An error occurred while trying to create mutex (sys error message - No space left on device)

Root Causes:

  • Disk space exhausted on Load Generator
  • System resources (memory, file handles) exhausted
  • LoadRunner Agent configuration changed unexpectedly
  • Network configuration changes
  • Security patches or updates applied

Resolution Steps:

  1. Check system resources: Verify disk space, memory, and CPU on Load Generator
  2. Review recent changes: Check if any system updates, patches, or configuration changes were applied
  3. Restart Agent: Stop and restart the LoadRunner Agent process/service
  4. Check Agent Runtime Settings: Go to LoadRunner Agent Runtime Settings and verify "Allow Vusers without user login" option
  5. Reboot Load Generator: If issue persists, reboot the Load Generator machine

2. Script Upload & Deployment Issues

Script Upload Failed: Failed Downloading ALM Client HIGH

Symptoms:

  • Error: "Failed Downloading ALM Client. Please check network connection and user network credentials"
  • Script upload fails when trying to upload to LoadRunner Enterprise
  • Error appears during ZIP file upload process

Root Causes:

  • OTA (Open Test Architecture) client not initialized correctly on LoadRunner Enterprise server
  • Network connectivity issues between client and LoadRunner Enterprise server
  • User credentials issues
  • LoadRunner Enterprise server configuration problems

Resolution Steps:

  1. Check server logs: Review LoadRunner Enterprise Server log.txt file in <PC Server>\orchidtmp\LTLogger\HP.PCLTOP.QCOTAOperatioinServiceWrapper
  2. Verify OTA client: Ensure OTA client is properly installed and configured
  3. Check network connectivity: Verify network connection between workstation and LoadRunner Enterprise server
  4. Verify credentials: Ensure user has proper permissions in LoadRunner Enterprise
  5. Restart LoadRunner Enterprise services: Restart relevant LoadRunner Enterprise services on the server
‚ö†Ô∏è Escalate to Core Team if: OTA client issues require server-side reconfiguration or if LoadRunner Enterprise services need restart

Getting "Only Zipped VuGen" Error When Uploading Script MEDIUM

Symptoms:

  • Error message indicates only zipped scripts are accepted
  • Upload fails even though script is zipped

Root Causes:

  • Script was not properly zipped from the correct root directory
  • ZIP file structure is incorrect (script folder should be at root of ZIP)
  • ZIP file is corrupted
  • ZIP file contains extra folders or files not part of the script

Resolution Steps:

  1. Verify ZIP structure: Open the ZIP file and ensure the script folder (containing .usr, .prm, data files, etc.) is at the root level
  2. Re-zip correctly: Navigate to the parent folder of your script folder, select the script folder, and create a new ZIP
  3. Check file contents: Ensure all necessary script files (.usr, .c files, data files, .prm) are included
  4. Test ZIP file: Extract the ZIP to a temporary location and verify it can be opened in VuGen before uploading

REST API Script Upload Returns Error 1001 MEDIUM

Symptoms:

  • HTTP Status Code 500 (Internal Server Error)
  • Error Code: 1001, Exception Message: "Operation failed"
  • Occurs when uploading script via REST API

Root Causes:

  • Incorrect REST API syntax or parameters
  • Authentication issues
  • Project or domain path incorrect
  • Script format not compatible

Resolution Steps:

  1. Verify REST API endpoint: Ensure correct URL format: http://{server}/LoadTest/rest/domains/{domain}/projects/{project}/Scripts
  2. Check authentication: Verify authentication token or Basic Auth credentials are correct
  3. Verify HTTP method: Use POST method for script upload
  4. Check script format: Ensure script is properly zipped before upload
  5. Test with Postman: Use Postman or similar tool to test API call independently

Script Works in VuGen But Fails When Deployed from Controller HIGH

Symptoms:

  • Script runs successfully in VuGen
  • Same script fails when executed from Controller or LoadRunner Enterprise
  • May see parameter file errors or missing data file errors

Root Causes:

  • Parameter files using different case sensitivity
  • File paths not correctly referenced
  • Data files not included in script upload
  • Runtime settings differences between VuGen and Controller

Resolution Steps:

  1. Check parameter file names: Ensure all parameter file references use consistent case (LoadRunner Enterprise is case-sensitive)
  2. Verify .prm file: Check that ScriptName.prm file references parameters with correct case
  3. Remove spaces from file names: Ensure no spaces in parameter file names (spaces work in VuGen but not LoadRunner Enterprise)
  4. Verify TableLocation: If TableLocation in .prm is not "Local", ensure parameter file exists in specified path
  5. Include all data files: Ensure all external data files are included in the script ZIP
  6. Check runtime settings: Compare runtime settings between VuGen and Controller scenario

3. Test Execution & Timeslot Issues

Symptoms:

  • Timeslot was scheduled for autostart but test run did not begin
  • Timeslot shows as complete but no test execution occurred
  • No error message visible in UI

Root Causes:

  • Timeslot became invalid due to host allocation problems
  • License issues prevented test from starting
  • Performance test became invalid (script changes, allocation errors)
  • Autostart option was not properly enabled
  • System stopped trying to run after autostart window expired

Resolution Steps:

  1. Open timeslot: In Timeslots module, double-click the timeslot to view failure reason
  2. Check autostart setting: Verify "Automatically" option was checked when reserving timeslot
  3. Review host allocation: Check if all required hosts were available during timeslot window
  4. Verify test validity: Open the test in Test Management and check for errors in design page
  5. Check license availability: Verify project has sufficient license allocation
  6. Calculate availability: Click "Calculate Availability" on the timeslot to see if it's valid
‚ö†Ô∏è Escalate to Core Team if: Host allocation issues require infrastructure changes or license adjustments are needed

Root Causes:

  • Changes in host properties or status

  2. Review Event Logs: Click the host and check Event Logs for PC_hostChangeStateSuccess and PC_hostChangeStatusSuccess events
  3. Verify host pools: Ensure required hosts are assigned to correct pools
  4. Check pool assignments: Verify pools are assigned to your project
  5. Calculate availability: Try to recalculate timeslot availability to see if system can reshuffle hosts
‚ö†Ô∏è Escalate to Core Team if: Host configuration changes are needed or hosts are down and require infrastructure team intervention

4. Result Collation & Analysis Issues

Failed to Collate Results - Timeout Exceeded HIGH

Symptoms:

  • Error: "Failed to collate results. Reason: 'Failed to collate. Timeout exceeded'"
  • Typically occurs after long-duration tests (8+ hours, 24+ hours)
  • Collation process starts but never completes

Root Causes:

  • Default collate timeout (5 minutes) insufficient for large result sets
  • Very large test results requiring more processing time
  • Network delays between load generators and LoadRunner Enterprise server
  • Connectivity issues during collation process

Resolution Steps:

  1. Increase collate timeout:
    • Login to project at http://<LRE_Server>:8080/qcbin
    • Go to Tools > Performance Center Controller Options > Run Results > Collate Settings > Timeout
    • Increase "Collate timeout in minutes" to a larger value (e.g., 30 or 60 minutes)
  2. Retry collation: After increasing timeout, retry the collation process
  3. Manual collation: If automatic collation continues to fail, try manual collation from Controller
  4. Check network: Verify stable network connectivity between load generators and server
üí° Prevention: For long-duration tests (12+ hours), proactively increase the collate timeout before running the test.
‚ö†Ô∏è Escalate to Core Team if: Collation continues to fail even with increased timeout, or if server resources are insufficient for processing large result sets

Analysis Error: Process HP.PC.AnalysisWrapper Abnormally Terminated HIGH

Symptoms:

  • Error: "Result analysis critical error. Reason: 'Process HP.PC.AnalysisWrapper abnormally terminated'"
  • Occurs after collation completes, during analysis phase
  • Analysis process crashes before completing

Root Causes:

  • Insufficient memory on LoadRunner Enterprise server or Analysis machine
  • Very large result sets overwhelming Analysis process
  • Corrupted result data
  • Analysis timeout too short for large datasets

Resolution Steps:

  1. Check server resources: Verify available memory and disk space on LoadRunner Enterprise server
  2. Increase analysis timeout: Adjust analysis timeout settings in project configuration
  3. Download results locally: Download raw results and try analyzing locally using Analysis application
  4. Break down analysis: Try analyzing smaller time ranges of the test if possible
  5. Check result integrity: Verify no .eve files are missing from load generator results
‚ö†Ô∏è Escalate to Core Team if: Server memory or resources need to be increased, or if Analysis server configuration requires changes

EVE File Missing from Load Generator Causing Collation Failure HIGH

Symptoms:

  • Automatic collation fails
  • Manual collation from raw results also fails
  • When checking raw results in Controller orchidtemp folder, .eve file from one Load Generator is missing
  • Incomplete data from specific load generator

Root Causes:

  • Load Generator crashed or stopped responding during test
  • Disk space exhausted on Load Generator during test execution
  • Network interruption prevented file transfer
  • Load Generator process terminated prematurely

Resolution Steps:

  1. Check Load Generator logs: Review logs on the Load Generator that has missing .eve file
  2. Verify disk space: Check if Load Generator ran out of disk space during test
  3. Review test execution logs: Check Controller logs for any Load Generator errors during test
  4. Partial analysis: If one LG's data is missing, analyze results from other LGs only (partial results)
  5. Document for future: Identify root cause to prevent recurrence in next test
‚ö†Ô∏è Escalate to Core Team if: Load Generator infrastructure issues need to be resolved, or if test needs to be re-run

6. Script Replay Errors

Error -26612 HTTP Status Code 403 (Forbidden) HIGH

Symptoms:

  • Script replay fails with Error -26612
  • HTTP response shows 403 Forbidden status
  • Script may have worked during recording but fails during replay

Root Causes:

  • Authentication failure - session not properly maintained
  • Missing or incorrect correlation of security tokens
  • Server rejecting request due to invalid credentials
  • IP restrictions on server side
  • Missing required headers or cookies

Resolution Steps:

  1. Review authentication: Verify login credentials are correct and properly parameterized
  2. Check correlation: Look for dynamic values that need to be correlated (session IDs, tokens, etc.)
  3. Compare recording vs replay: Use "Recording & Replay Comparison" in VuGen to identify differences
  4. Add correlation rules: Correlate all dynamic values found in responses
  5. Verify headers: Ensure all necessary headers are being sent
  6. Check cookies: Verify cookie handling is enabled and working correctly
üí° Tip: Use VuGen's automatic correlation feature first, then manually review for missed correlations

Error -27732 Internal Error - Can't Parse Username MEDIUM

Symptoms:

  • Error: "Internal Error - can't parse username"
  • Or: "Internal Error - can't get initial credentials for username/password"
  • Occurs during script replay, particularly with integrated authentication

Root Causes:

  • Integrated Windows authentication not configured properly
  • Credentials in web_set_user function are invalid or expired
  • Authentication scheme mismatch

Resolution Steps:

  1. Verify credentials: Check that username and password in web_set_user are still valid
  2. Check authentication type: Verify correct authentication scheme (NTLM, Basic, Digest, etc.)
  3. Update web_set_user: Ensure correct syntax: web_set_user("username", "password", "host:port");
  4. Test authentication manually: Verify credentials work in browser
  5. Check domain requirements: For Windows auth, may need to include domain: "DOMAIN\\username"

SSL Protocol Error HIGH

Symptoms:

  • Error: "SSL protocol error when attempting to connect with host"
  • Error: "SSL State: error in SSLv3 write client certificate"
  • Occurs when replaying scripts with HTTPS/SSL

Root Causes:

  • SSL certificate issues
  • Certificate not properly imported into LoadRunner
  • Certificate hierarchy incomplete
  • SSL version mismatch

Resolution Steps:

  1. Import certificates: Add all certificates in the certificate hierarchy to a .pem file
  2. Configure SSL settings: In Runtime Settings > Network > SSL:
    • Enable SSL
    • Point to .pem file location
    • Configure SSL version if needed
  3. Disable webpage breakdown: If using HTTPS with proxy server:
    • In Runtime Settings > Internet Protocol > Preferences
    • Uncheck "Enable webpage breakdown"
  4. Try WinInet: If recorded with WinInet, use WinInet Replay Engine in Runtime Settings

Error -27792 Failed to Connect to Server - Connection Refused MEDIUM

Symptoms:

  • Error: "Failed to connect to server [server:port] [10061] Connection Refused"
  • Script cannot establish connection to target server

Root Causes:

  • Target server is down or unreachable
  • Firewall blocking connection
  • Port is closed or not listening
  • Network routing issues
  • Server hostname resolution failure

Resolution Steps:

  1. Verify server is up: Confirm target application is running and accessible
  2. Test connectivity: Try pinging server and telnet to specific port
  3. Check DNS resolution: Verify hostname resolves to correct IP address
  4. Check firewall: Ensure firewall rules allow connection from Load Generator
  5. Verify URL: Confirm server URL and port are correct in script

Transaction Not Found Error LOW

Symptoms:

  • Error message indicates transaction not found
  • Results show missing transactions
  • Transaction names appear incorrect in Controller

Root Causes:

  • lr_start_transaction and lr_end_transaction mismatch
  • Transaction name typo or inconsistency
  • Missing end transaction statement
  • Code logic error causing transaction to not execute

Resolution Steps:

  1. Verify transaction names: Check that lr_start_transaction and lr_end_transaction use exact same name
  2. Check for missing end: Ensure every start transaction has corresponding end transaction
  3. Review code flow: Verify transaction code path executes under test conditions
  4. Check nesting: Ensure transactions are not incorrectly nested

  4. Source the file: Run source ~/.bashrc to apply changes
  5. Re-run verify: Execute verify_generator.sh again
üí° Note: The verify_generator.sh script should be run after LoadRunner installation is complete on Linux

8. Firewall & Network Issues

Running Vusers Over Firewall - Decryption Failed HIGH

Symptoms:

  • Error: "Internal error - failed to read SL layer of a message"
  • Error: "Internal error - decryption failed at CSecurityLayerEncryptMessageReader:HandleMessage"
  • Occurs when using MI_Listener and running Vusers over firewall
  • Connection establishes but immediately fails with decryption error

Root Causes:

  • Mismatch in security settings between Load Generator and MI_Listener
  • Certificate/key configuration errors
  • "Check Client Certificates" or "Private Key Password" settings don't match
  • Secure connection configuration mismatch

Resolution Steps:

  1. Verify MI_Listener config: Check "Check Client Certificates" and "Private Key Password" in MI_Listener configuration
  2. Match LG settings: Ensure Load Generator Agent configuration has matching "Check Server Certificates" and "Private Key Password"
  3. Test without security: Temporarily disable "Use Secure Connection" to verify basic connectivity
  4. Verify versions: Ensure Load Generator, MI_Listener, and Controller are all same version
  5. Regenerate certificates: If using secure connection, regenerate and redistribute certificates
‚ö†Ô∏è Escalate to Core Team if: Certificate regeneration or MI_Listener reconfiguration is required

Multiple NICs Causing Controller-Load Generator Connectivity Issues HIGH

Symptoms:

  • Intermittent connection failures between Controller and Load Generator
  • Messages appear to be "lost" or not acknowledged
  • Time-out errors occur randomly
  • Controller thinks host is not responding

Root Causes:

  • Machine has multiple Network Interface Cards (NICs)
  • Messages sent from one NIC but responses expected on another
  • Communication gets "lost" due to incorrect interface routing
  • Agent binds to wrong network adapter

Resolution Steps:

  1. Identify primary NIC: In Windows, go to Network Connections > Advanced > Advanced Settings > Adapters and Bindings
  2. Reorder adapters: Move the primary adapter to top of the list
  3. Disable extra NICs: If possible, disable unused network adapters during testing
  4. Configure specific NIC: In LoadRunner Agent configuration, specify which NIC to use for communication
  5. Update hosts file: Add explicit IP-to-hostname mappings for the correct NIC
  6. Test connectivity: Verify bidirectional communication using the primary interface
üí° Best Practice: For dedicated Load Generators, disable or remove unused network adapters to avoid routing confusion

Port 54345 Already In Use HIGH

Symptoms:

  • LoadRunner Agent fails to start
  • Error indicates port 54345 is already bound
  • Cannot connect to Load Generator even though Agent appears to be installed

Root Causes:

  • Another application using port 54345
  • Previous LoadRunner Agent process did not terminate cleanly
  • Zombie process holding the port

Resolution Steps:

  1. Check port usage: Run netstat -ano | findstr 54345 (Windows) or netstat -an | grep 54345 (Linux)
  2. Identify process: Run netstat -anob (Windows with admin rights) to see which process is using the port
  3. Kill process: Terminate the process using the port (if safe to do so)
  4. Restart Agent: Start LoadRunner Agent service again
  5. Reboot if necessary: If port remains occupied, reboot the machine

Firewall Blocking Required Ports HIGH

Symptoms:

  • Cannot connect Controller to Load Generator despite network connectivity
  • Ping works but LoadRunner connection fails
  • Connection times out

Root Causes:

  • Firewall blocking LoadRunner communication ports
  • Windows Firewall or corporate firewall rules preventing connection
  • Ports 50500, 54345, or 443 (for firewall mode) blocked

Resolution Steps:

  1. Test port connectivity: Use telnet to test ports: telnet [hostname] 54345
  2. Check Windows Firewall: Verify LoadRunner exceptions exist in Windows Firewall
  3. Add firewall rules: Create inbound rules allowing traffic on:
    • Port 50500 (TCP)
    • Port 54345 (TCP)
    • Port 443 (TCP) - if using MI_Listener
  4. Check corporate firewall: Verify no network-level firewall blocking these ports
  5. Test after changes: Retry connection after firewall rules are added
‚ö†Ô∏è Escalate to Core Team if: Firewall changes require network security team approval or corporate firewall modifications

9. Java & Protocol-Specific Issues

Error 22994 Unable to Create Java VM HIGH

Symptoms:

  • Error: "22994: Error: Unable to create Java VM"
  • Java Vuser scripts fail to execute
  • Occurs during script compilation or replay

Root Causes:

  • PATH environment variable not properly configured for Java
  • CLASSPATH environment variable missing required entries
  • JDK not properly installed or configured
  • Conflicting Java installations

Resolution Steps:

  1. Set PATH: Ensure these directories are in PATH:
    • [JDK install dir]\bin
    • [LoadRunner install dir]\bin
  2. Set CLASSPATH: Ensure these are in CLASSPATH (preferably at beginning):
    • [LoadRunner install dir]\classes
    • [LoadRunner install dir]\classes\srv
    • [JDK install dir]\jre\lib\rt.jar
  3. Add application CLASSPATHs: Include any necessary application class paths
  4. Remove conflicts: Remove conflicting entries (e.g., QTP Java environment)
  5. Restart LoadRunner: Close and restart VuGen/Controller for changes to take effect
  6. Apply to Load Generators: If running remotely, apply same changes to all Load Generators and restart agents

Error 22992 Can't Find Actions.class MEDIUM

Symptoms:

  • Error: "22992 Can't find Actions.class"
  • Java script fails to execute
  • Actions.class file missing from script directory

Root Causes:

  • Java compilation failed - Actions.class not generated from Actions.java
  • CLASSPATH or PATH issues preventing compilation
  • Function within Actions class exceeds JVM 56KB limit
  • Compilation errors in Actions.java

Resolution Steps:

  1. Check if file exists: Look for Actions.class in script directory
  2. If missing, fix environment: Follow Error 22994 resolution steps
  3. Recompile script: In VuGen, use Program > Compile to force recompilation
  4. Check for function size: If function is too large (>56KB):
    • Break large functions into multiple smaller functions
    • Refactor code to reduce function size
  5. Clean and rebuild:
    • Delete old initialization files
    • Run register_vugen.bat from LoadRunner bin directory
    • Backup and delete vugen.ini from Windows directory
    • Restart VuGen

Multiple JDK Versions Required on Same Load Generator MEDIUM

Symptoms:

  • Need to run scripts requiring different JDK versions concurrently
  • Cannot configure single Load Generator to support multiple JDKs simultaneously

Root Causes:

  • Different applications/scripts require different Java versions
  • LoadRunner Agent can only use one JDK at a time

Resolution Steps:

  1. Use Terminal Services approach:
    • Launch LoadRunner Agent as a process (not service)
    • Enable Terminal Services in LoadRunner: Start > Programs > LoadRunner > Advanced Settings > Agent Configuration > Enable Terminal Services
    • Open different terminal sessions for each JDK environment
    • Launch separate Agent instances in each terminal session with different JDK paths
  2. Alternative: Multiple Load Generators: Use separate Load Generator machines for different JDK requirements

JDK Not Found Error in LoadRunner 12.62+ MEDIUM

Symptoms:

  • Error: "JDK not found"
  • LoadRunner cannot locate installed JDK
  • Java scripts fail to run

Root Causes:

  • JDK not installed
  • LoadRunner not configured with JDK path
  • Incorrect JDK version
  • Environment variables not set

Resolution Steps:

  1. Verify JDK installation: Confirm JDK (not just JRE) is installed
  2. Configure JDK path: In VuGen, go to Tools > Options > Environment and set JDK path
  3. Set JAVA_HOME: Set JAVA_HOME environment variable to JDK installation directory
  4. Verify version compatibility: Ensure JDK version is compatible with LoadRunner version
  5. Restart VuGen: Close and restart VuGen after configuration changes

10. Infrastructure & Resource Issues

Sessions Hanging in Server After LoadRunner Test Completes MEDIUM

Symptoms:

  • LoadRunner test completes but application sessions remain open
  • Server Manager shows sessions from LoadRunner user still active
  • Sessions not properly logged out or cleaned up

Root Causes:

  • Script does not include proper logout process
  • Script failures prevent logout from executing
  • Incomplete session cleanup in application
  • Script errors cause Vusers to terminate without cleanup

Resolution Steps:

  1. Add logout to script: Ensure vuser_end section includes complete logout process
  2. Add browser close: Include "close browser" at start of script to cleanup previous sessions
  3. Add contingent logout: Implement error handling that performs logout even if script fails
  4. Use try-finally blocks: Wrap main actions in try-finally to ensure cleanup code executes
  5. Verify logout completion: Add verification that logout completed successfully
üí° Best Practice: Always include cleanup code in vuser_end() section, and use error handling to ensure cleanup runs even if script fails

Process mmdrv.exe Not Created on Remote Host HIGH

Symptoms:

  • Error: "Process 'mmdrv.exe' was not created on remote host"
  • Reason: "communication error"
  • Agent process/service is running but process cannot be created

Root Causes:

  • Communication error between Controller and Load Generator
  • Port binding issues (same as Error -10344)
  • Insufficient permissions to create processes
  • Antivirus or security software blocking process creation

Resolution Steps:

  1. Check Agent log: Review LoadRunner_agent_startup.log for errors
  2. Verify port availability: Follow Error -10344 resolution steps
  3. Check permissions: Ensure LoadRunner Agent has rights to create processes
  4. Review antivirus: Add LoadRunner directories to antivirus exclusion list
  5. Verify disk space: Ensure sufficient disk space on Load Generator
  6. Restart Agent: Stop and restart LoadRunner Agent service

Insufficient Memory on Load Generator HIGH

Symptoms:

  • Out of memory errors during test execution
  • Vusers fail to initialize or crash randomly
  • Load Generator becomes unresponsive
  • Test cannot reach target user load

Root Causes:

  • Too many Vusers allocated to single Load Generator
  • Insufficient RAM on Load Generator machine
  • Memory leaks in scripts
  • Inefficient script design consuming excessive memory

Resolution Steps:

  1. Monitor resources: Use Task Manager/Resource Monitor to check memory usage
  2. Reduce Vuser count: Distribute Vusers across more Load Generators
  3. Optimize scripts: Review scripts for memory leaks or inefficient memory usage
  4. Enable multithreading: In Runtime Settings, enable "Run Vuser as a thread" to reduce memory footprint
  5. Increase RAM: Add more RAM to Load Generator if possible
  6. Close unnecessary applications: Stop non-essential services on Load Generator
‚ö†Ô∏è Escalate to Core Team if: Additional Load Generators need to be provisioned or hardware upgrades are required

Load Generator Running Out of Disk Space During Test HIGH

Symptoms:

  • Test fails mid-execution
  • Error messages indicate disk full
  • Result files incomplete or corrupted
  • Cannot connect to Load Generator after test starts

Root Causes:

  • Large log files generated during test
  • Extensive logging enabled in Runtime Settings
  • Old test results not cleaned up
  • Insufficient disk space allocated to partition

Resolution Steps:

  1. Check available space: Monitor disk space before starting tests
  2. Reduce logging: In Runtime Settings > Log:
    • Set logging level to "Brief" or "Send messages only when error occurs"
    • Disable extended logging unless debugging
  3. Clean temp directories: Delete old files from LoadRunner temp folder
  4. Archive old results: Move or delete results from previous test runs
  5. Allocate more space: Increase disk space if tests regularly run out
üí° Prevention: Calculate expected result size (approximately 1-2 MB per Vuser-hour) and ensure 2-3x that space is available before starting tests

Network Latency Issues During Test Execution MEDIUM

Symptoms:

  • Unusually high response times
  • Timeouts during test execution
  • Results inconsistent with application performance
  • Load Generators report network errors

Root Causes:

  • Network congestion between Load Generators and application
  • Firewall or proxy introducing latency
  • DNS resolution delays
  • Network infrastructure issues

Resolution Steps:

  1. Measure actual network latency: Use ping and traceroute to measure latency
  2. Check network utilization: Monitor network bandwidth usage during test
  3. Use DNS cache: In Runtime Settings > DNS Caching, enable DNS caching
  4. Adjust timeouts: Increase timeout values in Runtime Settings if network latency is expected
  5. Use connection pooling: Enable "Simulate browser cache" and "Keep-Alive" to reduce connection overhead
  6. Test from different location: Try running from Load Generators closer to application

Controller Becomes Unresponsive During Large Test HIGH

Symptoms:

  • Controller UI freezes or becomes very slow
  • Cannot interact with Controller during test execution
  • High CPU usage on Controller machine
  • Memory usage continuously increasing

Root Causes:

  • Too many Vusers being managed by single Controller
  • Excessive monitoring and graphing
  • Controller machine under-spec'd for test size
  • Real-time results updating overwhelming system

Resolution Steps:

  1. Reduce real-time monitoring: Minimize number of open graphs during test
  2. Disable auto-refresh: Turn off automatic refresh of graphs during execution
  3. Reduce granularity: Increase polling interval for monitors
  4. Use LoadRunner Enterprise: For very large tests, use LoadRunner Enterprise instead of standalone Controller
  5. Upgrade Controller hardware: Ensure Controller machine meets requirements for test size
  6. Distribute load: Split tests across multiple scenarios if possible
